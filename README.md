## Visión general

Este servicio gestiona el ciclo de vida de usuarios a través de una arquitectura hexagonal limpia. Expone una API REST que persiste en PostgreSQL y publica eventos Avro a Kafka para que otros servicios reaccionen.

Principios de diseño:
- **Schema-first**: todos los mensajes Kafka se validan contra schemas Avro registrados en Confluent Schema Registry
- **Arquitectura limpia**: la lógica de dominio está completamente desacoplada de la infraestructura
- **Extensible**: construido para escalar a un ecosistema completo de microservicios con múltiples bounded contexts

## Arquitectura

El servicio sigue **Arquitectura Hexagonal (Ports & Adapters)**:

```
common/
  application/     — Mediator, Command, CommandHandler, VoidResponse (genéricos)
  infrastructure/  — KafkaProducer, configuración de Kafka, utilidades de mensajería

user/
  domain/          — entidades (User, Role), eventos de dominio, puertos (UserRepository, UserEvent), excepciones
  application/
    http/          — Commands y Handlers disparados desde HTTP (create, delete, update)
    kafka/         — Commands y Handlers para eventos Kafka entrantes (created, deleted, updated)
  infrastructure/
    api/           — UserController, DTOs de request/response, UserApiMapper
    database/      — UserEntity, UserRepositoryImpl, UserRepositoryData (Spring Data JPA), UserEntityMapper
    event/
      producer/    — UserEventProducer (implementa el puerto UserEvent)
      consumer/    — UserKafkaEventConsumer + SpecificConsumers por tipo de evento

autogenerated/     — Clases Java generadas automáticamente desde schemas Avro (NO editar a mano)
```

### Flujo HTTP → Kafka

```
UserController → Mediator.dispatch(Command) → *Handler → UserRepository (PostgreSQL) + UserEvent (publica Avro en Kafka)
```

### Patrón Mediator

El `Mediator` descubre automáticamente todos los beans `CommandHandler` en el contexto y los indexa por `getCommandType()`. Para añadir un nuevo caso de uso, basta con crear un `Command` y su `CommandHandler` anotado con `@Component`.

## Primeros pasos

### Prerrequisitos

- Java 21+
- Docker & Docker Compose
- Maven (o usar `./mvnw` incluido)

### 1. Configurar variables de entorno

```bash
cp .env.example .env
```

Edita `.env` y rellena los valores, especialmente `DB_PASSWORD`.

### 2. Levantar infraestructura local

```bash
docker compose up -d
```

Levanta: Zookeeper, Kafka Broker, Schema Registry, Confluent Control Center y PostgreSQL.

| Servicio              | URL                          |
|-----------------------|------------------------------|
| Confluent Control Center | http://localhost:9021      |
| Schema Registry       | http://localhost:8081        |
| PostgreSQL            | localhost:5432               |

### 3. Compilar el proyecto

```bash
./mvnw clean install
```

> Las clases Java de Avro se auto-generan desde los `.avsc` durante la compilación.

### 4. Ejecutar la aplicación

```bash
./mvnw spring-boot:run
```

La API estará disponible en `http://localhost:8080`.

## API REST

Base path: `/api/v1/users`

### Crear usuario

```http
POST /api/v1/users/create
Content-Type: application/json

{
  "firstName": "Sergio",
  "lastName": "Ricart",
  "email": "sergio@example.com",
  "password": "secret123",
  "role": "USER"
}
```

Respuesta: `201 Created`

### Eliminar usuario

```http
DELETE /api/v1/users/delete/{id}
```

Respuesta: `204 No Content`

### Actualizar usuario

```http
PATCH /api/v1/users/update/{id}
Content-Type: application/json

{
  "firstName": "Sergio",
  "lastName": "Ricart",
  "email": "nuevo@example.com",
  "password": "nuevaPassword",
  "role": "ADMIN"
}
```

Respuesta: `204 No Content`

## Eventos Kafka

Topic principal: **`user.events`**

Todos los eventos se serializan en Avro y se registran en Schema Registry. Los schemas fuente están en `src/main/resources/schemas/user/`.

| Evento              | Campos                                                          |
|---------------------|-----------------------------------------------------------------|
| `UserCreatedEvent`  | `firstName`, `lastName`, `email`, `role`, `password`, `timestamp` |
| `UserUpdatedEvent`  | `id`, `firstName`, `lastName`, `email`, `role`, `password`, `timestamp` |
| `UserDeletedEvent`  | `id`, `timestamp`                                               |

La especificación AsyncAPI completa está en `src/main/resources/asyncapi.yml`.

## Configuración

Las variables de entorno se cargan desde `.env` (vía `spring-dotenv`). Valores configurables:

| Variable                 | Por defecto             | Descripción                        |
|--------------------------|-------------------------|------------------------------------|
| `DB_HOST`                | `localhost`             | Host de PostgreSQL                 |
| `DB_PORT`                | `5432`                  | Puerto de PostgreSQL               |
| `DB_NAME`                | `user_db`               | Nombre de la base de datos         |
| `DB_USER`                | `sricart`               | Usuario de la base de datos        |
| `DB_PASSWORD`            | —                       | Contraseña (obligatoria)           |
| `KAFKA_BOOTSTRAP_SERVERS`| `localhost:9092`        | Dirección del broker Kafka         |
| `SCHEMA_REGISTRY_URL`    | `http://localhost:8081` | URL del Schema Registry            |

## CI/CD

El pipeline de GitHub Actions se ejecuta automáticamente en cada **Pull Request hacia `main`**.

**Pasos:**
1. Checkout del código
2. Setup de Java 21 (Temurin) con caché de Maven
3. Levanta un servicio PostgreSQL real (Docker service)
4. Ejecuta `./mvnw clean test`

El pipeline **no mockea la base de datos** — los tests corren contra PostgreSQL real para garantizar que las migraciones y consultas funcionen en producción.
